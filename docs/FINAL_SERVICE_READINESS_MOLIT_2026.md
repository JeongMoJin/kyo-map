## 1. 총평

공가지도는 "아이디어 문서"가 아니라 실제 URL/로컬에서 실행되는 지도형 시제품이다. 지도, 필터, 후보 상세, 행정 대시보드, CSV 다운로드까지 이어지는 사용 흐름이 있어 제품/서비스 개발 부문 기본 요건은 충족한다. 특히 "등록 빈집 조회"가 아니라 "미등록 빈집 후보를 사전에 좁히고 우선조사 대상으로 추천"한다는 방향은 대회 주제와 잘 맞다.

그러나 현재 상태 그대로 최종심사에 올라가면 가장 큰 약점은 코드 완성도가 아니라 신뢰성이다. 화면은 실제 AI·실시간·한전 연동·전국 탐지를 하는 서비스처럼 보이지만, 레포 기준 실제 구현은 샘플 JSON과 산식 기반 MVP다. 발표자가 이를 먼저 정직하게 설명하지 않거나 화면 문구를 보정하지 않으면 심사위원 질문에서 크게 흔들릴 수 있다.

## 4. P0 문제

### P0-1. 실제 구현보다 강한 실시간/AI/연동 표현

- 문제: `LiveTicker`, `SiteHeader`, metadata, dashboard 일부 문구가 실제 연동 서비스처럼 보인다.
- 왜 치명적인가: 최종 발표심사에서 "지금 한전 데이터가 실시간 연동되나요?", "실제 AI 모델을 호출하나요?" 질문이 나오면 현재 코드와 답변이 충돌한다.
- 최소 대응: 발표 시작 30초 안에 "오늘 보여드리는 것은 샘플 데이터 기반 시제품이며, 실제 실증에서는 지자체 데이터와 공공데이터를 연동합니다"를 먼저 말한다.
- 이상 대응: UI 문구를 `실시간` -> `데모 시뮬레이션`, `AI 엔진 가동 중` -> `AI 분석 흐름 데모`, `자동 연계` -> `검토 흐름 연결 가능`으로 보정.

### P0-2. 공공데이터 원천-가공-서비스 사용 연결 증거 부족

- 문제: 데이터 출처는 UI에 있지만 실제 원본 파일, API 샘플, 가공 파이프라인 산출물이 없다.
- 왜 치명적인가: 제품/서비스 개발 부문에서 "공공데이터 활용 명확성"은 핵심 점수다.
- 최소 대응: 발표자료에 "현재 레포 데이터는 샘플, 실증 데이터는 건축물대장/전력/위성/안심구역/교통 접근성"으로 구분한 표를 넣는다.
- 이상 대응: 작은 원천 샘플 CSV 또는 익명화 가공 샘플, 데이터 딕셔너리, 파이프라인 로그를 추가한다.

### P0-3. AI 활용 증거 부족

- 문제: `GPT-4o`, `ViT`, `LSTM`은 화면/주석에 있지만 모델 추론 코드가 없다.
- 왜 치명적인가: AI 가점 10점을 노릴 수 있지만, 반대로 "AI가 장식인가?" 질문을 받기 쉽다.
- 최소 대응: "현재 구현은 AI 모델을 실시간 호출하는 것이 아니라, AI 분석 파이프라인의 산출 형태와 행정 의사결정 흐름을 구현한 MVP"라고 말한다.
- 이상 대응: 모델별 입력/출력 샘플, 추론 결과 JSON, 평가 계획, 최소 한 건의 실제 AI 분석 산출물을 문서화한다.
