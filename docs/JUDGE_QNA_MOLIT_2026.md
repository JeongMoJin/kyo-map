# 공가지도 최종심사 예상 질문 50개와 답변 초안

작성일: 2026-06-21
답변 원칙: 과장 금지, 확정 판정 금지, 현재 구현과 실증 계획 구분
최종 업데이트: 2026-06-21 실서비스 전환 구현 반영

## 0. 발표자가 먼저 외워야 할 업데이트 답변

1. 공공데이터를 실제로 썼나요?
   예. 현재 서버는 `data.go.kr`의 `경상북도 영주시_빈집 현황` 공개 CSV를 직접 수집합니다. 검증 결과 총 950건, 기준일 2025-07-10이며, 공개 데이터가 정확 지번을 제외하므로 공개 화면은 리 단위 후보군으로 표시합니다.

2. AI가 실제로 무엇을 하나요?
   후보 상세 화면에서 서버 API가 OpenAI Responses API를 호출해 빈집 후보 가능성, 위험도, 권장 조치, 현장점검 체크리스트, 데이터 한계를 구조화 JSON으로 생성하도록 구현했습니다. 현재 로컬 환경은 `OPENAI_API_KEY`가 없어 로컬 정책 산식 fallback을 보여줍니다.

3. 지금 화면의 fallback은 실패인가요?
   실패가 아니라 운영 안정성 장치입니다. 키가 없거나 API가 장애일 때도 현장조사 우선순위와 조치안 초안을 제공하고, 키가 설정되면 같은 화면에서 OpenAI 결과가 표시됩니다.

4. 왜 영주시 데이터인가요?
   공개 파일로 정확히 접근 가능한 실제 지자체 빈집 현황 데이터가 있어 1차 PoC로 사용했습니다. 국토부 실서비스로 확장할 때는 건축HUB, 법정동 코드, VWorld, 지자체 빈집대장과 같은 전국 단위 데이터로 확장합니다.

## 1. 공공데이터 관련

1. 어떤 공공데이터를 사용했습니까?
   현재 시제품은 건축물대장, 전력사용 패턴, 위성영상, 안심구역, 교통 접근성 데이터를 결합하는 구조를 샘플 데이터로 구현했습니다. 실제 실증 단계에서는 지자체 빈집대장과 공공데이터 원천을 연결해 검증할 계획입니다.

2. 레포에 실제 원본 공공데이터가 있습니까?
   현재 레포에는 원본 공공데이터 파일이 아니라 샘플 후보 100건과 생성 스크립트가 있습니다. 그래서 발표에서는 샘플 데이터 기반 MVP라고 명확히 말해야 합니다.

3. 건축물대장은 어떻게 반영됩니까?
   후보의 준공연도, 면적, 대장상 용도 같은 필드로 반영됩니다. 오래된 건물일수록 안전위험과 우선조사 필요도가 높아지는 식으로 산식에 들어갑니다.

4. 전력사용량은 어떤 의미입니까?
   최근 6개월 평균 전력사용량이 매우 낮게 지속되면 장기 미거주 가능성이 있다는 신호로 봅니다. 다만 전력만으로 빈집을 확정하지는 않고 현장 확인이 필요합니다.

5. 위성영상은 실제로 분석했습니까?
   현재 구현은 실제 위성영상 모델 추론까지 들어간 상태는 아닙니다. 화면은 위성영상 분석 결과가 들어왔을 때 어떤 형태로 후보 근거를 보여줄지 구현한 시제품입니다.

6. 안심구역 데이터는 실제 API로 연결되어 있습니까?
   현재는 `isDisasterZone` 샘플 플래그로 구현되어 있습니다. 실제 서비스에서는 안심구역 또는 재난/위험 관련 공간데이터와 좌표 중첩을 통해 반영해야 합니다.

7. 데이터 갱신 주기는 어떻게 설계합니까?
   건축물대장과 행정자료는 정기 갱신, 전력사용 패턴은 월 단위 또는 분기 단위, 현장확인 결과는 조사 완료 시점에 반영하는 구조가 현실적입니다.

8. 데이터 출처가 UI에 보입니까?
   대시보드와 필터/상세 화면에 데이터 출처 문구가 있습니다. 다만 현재는 원천/가공/샘플 여부가 충분히 분리되어 있지 않아 보완이 필요합니다.

9. 개인정보 문제는 없습니까?
   현재 샘플 데이터에는 개인 이름이나 연락처가 없습니다. 실제 실증에서는 주소와 소유자 정보가 민감할 수 있으므로 표시 수준을 낮추고, 가명/집계 처리와 권한 관리를 해야 합니다.

10. 공공데이터만으로 빈집 여부를 확정할 수 있습니까?
   확정할 수 없습니다. 공가지도는 현장조사 전 후보를 좁히는 도구이고, 최종 판단은 지자체 절차와 현장 확인을 통해 이루어져야 합니다.

## 2. AI 기술 관련

11. AI가 실제로 무엇을 합니까?
   현재 시제품은 AI 분석 파이프라인의 결과 형태와 행정 의사결정 흐름을 구현했습니다. 코드상 실시간 모델 호출은 없고, 후보 신뢰도와 우선순위 산식이 화면에 반영됩니다.

12. GPT-4o를 실제 호출합니까?
   현재 레포에는 GPT-4o API 호출이 없습니다. 발표에서는 GPT-4o는 용도 추천 설명을 생성할 수 있는 실증 단계 구성요소로 말하는 것이 안전합니다.

13. ViT와 LSTM 모델이 학습되어 있습니까?
   현재 레포에는 학습 코드나 모델 파일이 없습니다. ViT는 위성영상 분류, LSTM은 전력사용 시계열 학습이라는 목표 파이프라인으로 제시되어 있습니다.

14. 그러면 AI 활용이라고 부를 수 있습니까?
   현재 구현만 보면 실제 AI 추론보다는 AI 기반 제품 흐름의 MVP에 가깝습니다. AI 가점을 강하게 주장하려면 모델 추론 예시, 로그, 평가 설계가 추가로 필요합니다.

15. 우선순위 점수는 어떻게 계산합니까?
   `lib/priority.ts`에서 안전위험, 재생적합도, 현장확인 필요도, AI 신뢰도, 안심구역 여부, 건축연도, 전력사용량, 접근성을 가중치로 계산합니다.

16. 설명 가능성은 있습니까?
   있습니다. 상세 화면에는 AI 빈집 확률, 최근 전력 평균, 준공연도, 안심구역 또는 접근성 점수가 근거로 표시됩니다. 이 부분은 실제 행정 도구로서 장점입니다.

17. 정확도는 몇 퍼센트입니까?
   현재는 실증 전이라 정확도 숫자를 말하지 않는 것이 맞습니다. 지자체 빈집대장과 현장확인 결과를 기준으로 정밀도, 재현율, 우선조사 적중률을 검증할 계획입니다.

18. 오탐은 어떻게 처리합니까?
   오탐은 전제로 두어야 합니다. 후보 추천 후 지자체 빈집대장 매칭과 현장 확인을 거치고, 그 결과를 다시 점수 기준과 모델에 반영하는 루프가 필요합니다.

19. 사람이 직접 판단한 것과 AI 판단의 차이는 무엇입니까?
   사람은 개별 경험과 현장 정보에 강점이 있고, AI/스코어링은 넓은 후보군을 빠르게 좁히는 데 강점이 있습니다. 공가지도는 사람의 판단을 대체하지 않고 먼저 볼 대상을 정리합니다.

20. AI가 단순 장식이라는 비판에 어떻게 답합니까?
   현재 MVP에서는 실제 모델 호출은 부족합니다. 대신 후보 점수, 근거, 행정 조치 흐름을 구현했고, 실증 단계에서 실제 모델 산출물과 평가 지표를 붙이는 로드맵을 제시하겠습니다.

## 3. 기존 서비스 대비 차별성

21. 빈집애와 무엇이 다릅니까?
   빈집애는 공개·등록된 빈집 현황, 정비사업, 활용 사례, 매물 정보를 보여주는 역할이 강합니다. 공가지도는 등록 전 후보를 찾아 현장조사 우선순위를 제안하는 앞단 서비스입니다.

22. 기존 공식 시스템을 대체합니까?
   아닙니다. 기존 시스템을 대체하지 않고 실태조사와 빈집대장 관리의 앞단에서 후보를 좁히는 보조 도구입니다.

23. 한국부동산원 빈집정보시스템과 충돌하지 않습니까?
   충돌보다 보완 관계로 보는 것이 맞습니다. 공가지도는 현장조사 전 후보 추천과 정책 활용 시나리오를 웹 시제품으로 보여주는 데 집중합니다.

24. 기존에도 빈집 추정이 있는데 왜 필요합니까?
   기존 추정은 전기·수도 등 기준의 누락/과다추정 문제가 지적됩니다. 공가지도는 여러 데이터와 설명 가능한 우선순위를 결합해 조사 대상을 더 잘 정하는 방향을 제안합니다.

25. 공공기관이 왜 도입해야 합니까?
   조사 인력과 시간이 제한되어 있기 때문입니다. 위험도가 높거나 활용 가능성이 큰 후보를 먼저 보면 조사 효율과 정책 연결성이 좋아질 수 있습니다.

## 4. 제품/서비스 완성도

26. 실제 URL이 있습니까?
   README와 배포 보고서 기준 프로덕션 URL은 `https://kyo-map.vercel.app`입니다. 로컬에서도 `npm run start`로 실행 가능합니다.

27. 이 서비스는 지금 쓸 수 있습니까?
   시제품 데모로는 사용할 수 있습니다. 실제 행정 의사결정에 쓰려면 원천 데이터 연동, 권한 관리, 현장검증, 개인정보 보호 설계가 추가되어야 합니다.

28. 데모 흐름은 무엇입니까?
   지도에서 후보를 보고, 필터로 우선 대상을 좁히고, 상세에서 근거와 전력사용량을 확인한 뒤, 대시보드에서 Top10과 CSV를 보는 흐름입니다.

29. 모바일에서도 작동합니까?
   모바일 지도와 필터 바텀시트가 작동합니다. 다만 모바일 첫 화면에서 서비스 목적 설명이 짧아 발표용으로는 데스크톱 시연을 권장합니다.

30. 빌드는 통과했습니까?
   네. `npm run build`가 성공했고 `/`, `/dashboard`, `/house/[id]` 100개 상세 페이지가 정적으로 생성되었습니다.

31. 테스트는 있습니까?
   자동 테스트 스크립트는 현재 없습니다. 대신 이번 감사에서 TypeScript, build, dev/start, Playwright 수동 검증을 수행했습니다.

32. 장애 대응은 어떻게 합니까?
   로컬 서버, 핵심 화면 스크린샷, 30-60초 백업 영상을 준비해야 합니다. 외부 지도 타일 장애에 대비해 캡처 PDF도 필요합니다.

## 5. 정책 적합성/현장 적용

33. 실제 사용자는 누구입니까?
   1차 사용자는 지자체 빈집 실태조사, 건축정비, 재난안전, 지역경제 담당자입니다. 2차 사용자는 정책 기획자와 공공기관입니다.

34. 현장 담당자는 어떤 결정을 합니까?
   후보를 먼저 볼지, 위험 후보를 긴급 점검할지, 철거/정비/귀촌/창업 활용을 검토할지 결정합니다.

35. 정책 활용 가능성은 무엇입니까?
   빈집 실태조사, 정비계획, 매입 검토, 청년주거, 귀촌 지원, 로컬 창업공간, 소공원/마을쉼터 전환과 연결할 수 있습니다.

36. 전국 확장이 가능합니까?
   기술 구조는 가능하지만, 데이터 접근성과 지자체별 기준 차이가 핵심 변수입니다. 먼저 한 지자체 PoC로 검증한 뒤 확장하는 전략이 현실적입니다.

37. 특정 지역 PoC는 어디가 좋습니까?
   샘플 데이터와 문제 규모를 고려하면 경북, 전남, 강원 등 지방 빈집 문제가 큰 지역의 읍면 단위 실증이 적합합니다.

38. 유지보수 비용은 어떻게 봅니까?
   초기에는 정적/배치 분석과 대시보드 중심으로 비용을 낮출 수 있습니다. 실제 AI/위성/전력 연동까지 가면 데이터 처리와 검증 비용이 커집니다.

39. 지자체 업무시스템과 연동 가능합니까?
   현재는 연동되어 있지 않습니다. 향후에는 CSV, 내부망 업로드, API 연계, 권한 기반 대시보드 순서로 단계화하는 것이 현실적입니다.

40. 법적 빈집 판정과 충돌하지 않습니까?
   충돌하지 않도록 후보 추천으로 한정해야 합니다. 법적 판정은 현장조사와 행정 절차로 진행됩니다.

## 6. 법적/윤리적 리스크

41. 주소를 지도에 표시해도 됩니까?
   샘플 데모는 가능하지만 실제 서비스에서는 주소/좌표 표시 수준을 권한에 따라 제한해야 합니다. 일반 공개용과 행정 내부용을 분리하는 것이 안전합니다.

42. 소유자 정보는 다룹니까?
   현재는 소유자 이름이나 연락처를 다루지 않습니다. 실제 서비스에서도 소유자 개인정보는 직접 노출하지 않는 방향이 맞습니다.

43. 전력사용량은 민감하지 않습니까?
   민감할 수 있습니다. 개인 식별이 되지 않도록 가명/집계 처리하고, 목적과 권한을 명확히 해야 합니다.

44. AI 오판 책임은 누가 집니까?
   AI 결과는 참고 지표이고 최종 판단은 지자체 절차와 현장 확인으로 합니다. 서비스는 근거와 우선순위를 제공하는 보조 도구로 설계해야 합니다.

45. 저작권 문제는 없습니까?
   현재 상세 히어로는 `picsum.photos` 외부 이미지를 사용합니다. 발표자료에는 실제 빈집/위성 이미지처럼 보이지 않도록 주의하고, 가능하면 로컬/라이선스 확인 이미지로 교체해야 합니다.

## 7. 사업성/확장성/팀 역량

46. 수익모델은 무엇입니까?
   초기에는 지자체 PoC와 공공기관 실증 프로젝트가 현실적입니다. 이후 지역 단위 후보 탐지 리포트와 대시보드를 SaaS 또는 용역형으로 제공할 수 있습니다.

47. 수상 이후 계획은 무엇입니까?
   실제 공공데이터 샘플 확보, 지자체 빈집대장 매칭, 현장검증, AI 산출물 검증, 권한형 대시보드 고도화 순서로 추진합니다.

48. 왜 지금 필요합니까?
   미거주 주택은 증가하고 있고, 법령상 빈집과 통계상 미거주 주택 사이의 간극이 큽니다. 행정이 모든 후보를 직접 찾기 전에 우선순위를 정할 도구가 필요합니다.

49. 왜 이 팀이 할 수 있습니까?
   이미 Next.js 기반 지도, 상세, 대시보드, CSV까지 동작하는 시제품을 구현했습니다. 다만 실제 AI/공공데이터 실증은 후속 협력과 데이터 확보가 필요합니다.

50. 지금 가장 부족한 것은 무엇입니까?
   실제 원천 공공데이터와 AI 추론 증거입니다. 그래서 최종 발표 전에는 과장 표현을 줄이고, 실증 로드맵과 검증 설계를 명확히 제시해야 합니다.
