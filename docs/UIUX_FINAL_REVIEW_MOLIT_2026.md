# 공가지도 UI/UX 최종 리뷰

작성일: 2026-06-21
근거: Playwright 데스크톱/모바일 렌더링, 실제 스크린샷 수동 확인, 소스 코드
최종 업데이트: 2026-06-21 실서비스 전환 구현 반영

## 0. 실서비스 전환 후 UI/UX 판정

- 홈/대시보드에 `Production data` 패널이 추가되어 첫 화면에서 원천 데이터, 950건, 기준일, 키 상태가 보인다.
- 상세 화면에 `Real AI analysis` 패널이 추가되어 OpenAI/fallback 분석 결과를 사용자가 직접 실행할 수 있다.
- `실시간`, `GPT-4o`, `ViT/LSTM` 직접 실행처럼 보이던 과장 표현을 제거했다.
- Recharts 크기 경고를 제거했고, Playwright 검증 결과 브라우저 콘솔 경고/에러는 0건이다.
- 남은 UX 리스크는 운영 키 미설정 상태가 발표장에서 fallback으로 보인다는 점이다. 발표 전 실제 OpenAI 성공 화면을 캡처해야 한다.

## 1. 총평

지도와 대시보드는 "제품처럼 보이는" 수준까지 올라와 있다. 특히 지도-필터-상세-대시보드의 정보 구조는 심사위원이 서비스 흐름을 이해하기 좋다. 다만 최종 발표용 UI로는 세 가지가 약하다. 첫째, 첫 화면이 "미등록 후보 사전 스크리닝"이라는 핵심을 충분히 설명하지 못한다. 둘째, 일부 문구가 실제 구현보다 과장되어 신뢰 리스크가 있다. 셋째, 상세 페이지 히어로와 차트 경고 등 캡처 품질에 영향을 주는 문제가 있다.

## 2. 화면별 평가

| 화면 | 강점 | 약점 | 발표 적합성 |
| --- | --- | --- | --- |
| `/` 지도 | 실제 지도형 제품 느낌, 후보 마커/필터 명확, 모바일 대응 | 실시간/누적/한전 문구 과장, 핵심 목적 설명 부족 | 높음, 문구 보완 필요 |
| `/dashboard` | KPI, Official evidence, Top10, CSV가 행정 서비스처럼 보임 | "자동 연계", "실시간 운영" 표현 주의, 차트 콘솔 경고 | 높음 |
| `/house/[id]` | AI 분석/전력/행정 조치 흐름 좋음 | 히어로 텍스트 대비 낮음, `picsum` 이미지 실물성 약함 | 중간-높음 |
| 모바일 `/` | 지도와 필터 버튼 작동 | 첫 화면 정보가 압축되어 서비스 목적이 덜 보임 | 보조 데모용 |
| 404 | 안내와 복귀 버튼 있음 | 글로벌 404/에러 UI는 제한적 | 충분 |

## 3. 수정 제안

### 1. 첫 화면 목적 전달 부족

- 문제: `/` 첫 화면은 "전국 공가 100건"은 강하게 보이지만 "미등록 빈집 후보 사전 스크리닝"은 덜 보인다.
- 왜 심사에 불리한가: 심사위원이 5초 안에 기존 빈집 지도와 차별점을 이해해야 한다.
- 최소 수정안: 지도 오버레이 설명을 "미등록 빈집 후보를 현장조사 전 우선순위로 좁히는 데모"로 보강.
- 이상적인 수정안: 상단 카드에 `미등록 후보 탐지`, `우선조사 추천`, `정책 활용` 3개 짧은 배지 추가.
- 수정 예상 난이도: 낮음.
- 발표 전 우선순위: P1.

### 2. 실시간/AI/연동 표현 과장

- 문제: "LIVE", "실시간 탐지", "한전 데이터 연계 실시간", "AI 엔진 가동 중" 문구가 실제 구현과 불일치한다.
- 왜 심사에 불리한가: 신뢰성 문제로 번질 수 있다.
- 최소 수정안: 발표에서 먼저 "샘플 데이터 기반 MVP"라고 말하고, 화면 캡처에는 `데모` 표기를 추가.
- 이상적인 수정안: UI 문구를 `실시간`에서 `데모 시뮬레이션`, `AI 엔진 가동 중`에서 `AI 분석 흐름 데모`로 변경.
- 수정 예상 난이도: 낮음-중간.
- 발표 전 우선순위: P0.

### 3. 상세 페이지 히어로 텍스트 대비 부족

- 문제: 상세 페이지 주소 H1이 배경 이미지 위에서 충분히 읽히지 않는다.
- 왜 심사에 불리한가: 발표 캡처에서 후보 정보가 흐려지고 완성도가 낮아 보인다.
- 최소 수정안: 히어로 H1과 메타 텍스트에 전역 heading 색상보다 우선하는 흰색/그림자 스타일 적용.
- 이상적인 수정안: 히어로 전체를 실제 지도/위성형 정적 이미지 또는 중립 배경으로 바꾸고 정보 패널 대비를 확보.
- 수정 예상 난이도: 낮음.
- 발표 전 우선순위: P1.

### 4. 상세 이미지가 실제 빈집/위성이 아님

- 문제: `picsum.photos` 랜덤 이미지가 후보 주소/위성사진처럼 보인다.
- 왜 심사에 불리한가: 심사위원이 "실제 위성 이미지인가요?"라고 물으면 허술해 보인다.
- 최소 수정안: 발표에서는 상세 히어로를 오래 보여주지 않고 분석 카드/전력 차트로 바로 이동.
- 이상적인 수정안: 로컬 placeholder 또는 지도 기반 정적 캡처로 교체하고 "샘플 이미지" 표기.
- 수정 예상 난이도: 중간.
- 발표 전 우선순위: P1.

### 5. Recharts 크기 경고

- 문제: 콘솔에 chart width/height 경고가 여러 번 발생.
- 왜 심사에 불리한가: 특정 노트북/브라우저에서 차트가 잠시 비어 보일 수 있다.
- 최소 수정안: 데모 전 새로고침 후 차트가 안정적으로 보이는지 확인하고, 차트가 빈 경우 다른 화면으로 전환.
- 이상적인 수정안: 차트 wrapper에 명시적 크기와 mount 이후 렌더링 조건 추가.
- 수정 예상 난이도: 중간.
- 발표 전 우선순위: P1.

### 6. 모바일 첫 화면 정보 밀도

- 문제: 모바일에서는 설명문과 필터 정보가 많이 축약되어 서비스 목적이 약하다.
- 왜 심사에 불리한가: 심사위원이 휴대폰으로 URL을 열었을 때 "지도 앱" 정도로만 볼 수 있다.
- 최소 수정안: 모바일 오버레이에 `미등록 후보 사전 스크리닝` 한 줄을 유지.
- 이상적인 수정안: 모바일 첫 진입 시 짧은 목적 배너와 `대시보드 보기` CTA 제공.
- 수정 예상 난이도: 낮음-중간.
- 발표 전 우선순위: P2.

### 7. 시민 문의/지자체 제보 버튼의 데모성

- 문제: 상세 버튼은 토스트만 띄우지만 실제 접수처럼 보일 수 있다.
- 왜 심사에 불리한가: 기능 범위 질문에서 과장으로 보일 수 있다.
- 최소 수정안: 발표 중 버튼 클릭은 피하거나 "데모 토스트"라고 설명.
- 이상적인 수정안: 버튼 아래 `시제품 데모` 작은 라벨 추가.
- 수정 예상 난이도: 낮음.
- 발표 전 우선순위: P2.

### 8. 대시보드 색상/타이포

- 문제: 네이비/블루 계열이 강하고 일부 색상 대비가 강하다.
- 왜 심사에 불리한가: 공공기관 제안서 톤에는 맞지만, 한 화면에 파란 면적이 커서 캡처 시 무겁게 보일 수 있다.
- 최소 수정안: 현재 유지. 발표자료에서는 밝은 배경 슬라이드에 화면 캡처를 크게 배치.
- 이상적인 수정안: KPI 영역 높이를 조금 줄이고 Official evidence와 Top10이 더 빨리 보이게 조정.
- 수정 예상 난이도: 중간.
- 발표 전 우선순위: Nice-to-have.

## 4. 캡처 추천 화면

가장 좋은 발표 캡처:

1. `/` 데스크톱 지도 전체.
2. `/dashboard` 상단+Official evidence.
3. `/dashboard` Top10.
4. `/house/H00018` AI 분석 탭 중간부터, 히어로는 잘라내도 됨.
5. `/house/H00018` 전력사용량 탭.

피해야 할 캡처:

- 상세 히어로에서 H1이 어둡게 보이는 장면.
- 지도 타일이 아직 덜 로딩된 장면.
- 차트가 "준비 중" 또는 빈 상태인 장면.
- `picsum` 이미지가 실제 위성사진처럼 오해될 수 있는 클로즈업.
